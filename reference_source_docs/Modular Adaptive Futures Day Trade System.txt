1. SYSTEM PHILOSOPHY
Why Modular Beats Monolithic
A single end-to-end ML model trained to "predict the market" fails in production for a simple reason: markets are non-stationary. The statistical relationships that generate alpha in a trending environment actively destroy alpha in a mean-reverting one. A model that tries to learn both simultaneously learns neither cleanly — it smooths over the edges that define each regime and produces mediocre behavior everywhere.

The modular architecture solves this by decomposing the problem correctly. Each layer has a single, testable job:

Regime detection matters because market microstructure changes character throughout the day and across sessions. A 9:45 AM opening drive has different statistical properties than 12:30 PM midday chop or a 3:30 PM closing squeeze. Treating all bars as IID observations is the original sin of most retail trading systems. When you explicitly model regime, you can build strategies that are right for the conditions — not strategies that are on average slightly right across all conditions and actually terrible in any specific one.

Specialist strategies outperform a single generalist because their loss functions can be designed for a specific behavior. A trend pullback model should penalize missing continuation moves more than catching small reversals. A mean reversion model should be exactly the opposite. You cannot optimize for both simultaneously. Specialist models also fail in interpretable, diagnosable ways. When a trend model loses money during a range day, you know exactly why and exactly what to fix: the regime classifier failed to suppress it. The failure mode is isolated and actionable.

A policy selector improves robustness by creating an explicit governance layer between intelligence and action. Without a policy selector, you have a collection of models that can all fire simultaneously, conflict with each other, overload your position, or activate in the wrong environment. The policy selector treats regime output and specialist signals as inputs to a controlled decision process rather than direct execution commands. This is how institutional desks actually work — research generates signals, risk management decides what to act on.

A hard risk engine must sit above all models because models are probabilistic opinions and risk is arithmetic certainty. A model with 60% directional accuracy loses money if the losses on wrong trades are larger than gains on right trades. The risk engine does not know or care about model confidence. It knows your account balance, your daily loss, your position size, and your drawdown state. It enforces limits that no model output can override. This is not a preference — it is a structural requirement for surviving in futures.

This architecture is more realistic for NT8 + Python than an end-to-end black box for operational reasons. NT8 is a mature execution platform with excellent order management, native indicator calculation, and low-latency market data access. Python is where the serious numerical work belongs — scikit-learn, pandas, numpy, LightGBM, feature engineering pipelines, logging, and research tooling. Trying to do ML inside NT8 using C# is painful, fragile, and unmaintainable. Trying to do execution and order management in Python for futures is equally painful and introduces unnecessary latency and reliability risk. Give each environment the job it was designed for.

Avoiding Unstable Regime Flipping
Apply a regime persistence filter:

New regime only becomes "active" if it holds for 2+ consecutive bar classifications, OR if confidence exceeds 0.65
On transition, emit a "TRANSITION" state for one bar where size is halved
Do NOT allow regime to toggle back within 3 bars (minimum persistence = 3 bars)
Use an exponential moving average of regime probabilities with α = 0.3 to smooth the raw classifier output before policy decisions.

Best Modeling Approach
Recommendation: Gradient Boosting Classifier (LightGBM)

Why not alternatives:

Rule-based only: Too brittle, misses subtle regime signals, hard to calibrate across instruments
HMM: Theoretically elegant but assumes Markov property and Gaussian emissions — both questionable for real futures data
Neural nets: Require far more data, prone to overfit on regime labels, not interpretable
Random forest: Good but slower to train than LightGBM, slightly less accurate on tabular data
LightGBM wins because:

Handles tabular features natively, no normalization required
Trains in seconds, can be retrained daily
Outputs calibrated probabilities with predict_proba
Feature importance is directly interpretable
Regularization (num_leaves, min_data_in_leaf, lambda) prevents overfit
Proven on financial time-series classification tasks
Label the training data using forward-looking regime outcome (what type of day did this bar belong to, judged at end of session), with careful attention to point-in-time honesty. No future data should leak into features.

Recommended Specialist Set (Start With These)
SPEC-1: Trend Pullback Continuation Model
Purpose: Capture continuation entries during confirmed trend days when price pulls back to a dynamic support level.

Allowed regimes: TREND_UP, TREND_DOWN, BREAKOUT_CONTINUATION Forbidden regimes: RANGE_BOUND, MIDDAY_CHOP, NEWS_DISTORTED, RISK_OFF, NO_TRADE, BREAKOUT_FAILURE, MEAN_REVERSION

Features:

Distance from 20-EMA (pullback depth in ATR)
Distance from VWAP
Recent bar efficiency ratio
Volume on pullback bars (should be declining)
Prior swing high/low distance
Slope of EMA over last 8 bars
Type: Rule-based with ML confidence scoring. Simple rules define a valid setup; LightGBM confidence score gates entry.

Entry logic: Price pulls back to within 0.5–1.5 ATR of 20-EMA, efficiency ratio remains positive (>0.3), volume declining on pullback, then a single confirming bar closes in trend direction.

Invalidation: Price closes below 20-EMA (long) or above (short); volume surges on pullback (suggests reversal not continuation); regime flips to RANGE or CHOP.

Stop placement: Below the most recent pullback low (long) or above the most recent high (short). Stop width = 0.8–1.2 ATR. Never wider.

Target: 1.5× risk to first target, partial exit 50% of position. Trail remainder by 1 ATR from entry.

Aggressiveness: Moderate. Build this first. Most robust across instruments.

SPEC-2: Opening Range Breakout Confirmation Model
Purpose: Trade confirmed breakouts above/below the first 30-minute opening range when follow-through is evident.

Allowed regimes: OPENING_DRIVE, BREAKOUT_CONTINUATION, TREND_UP (for long breaks), TREND_DOWN (for short breaks) Forbidden regimes: RANGE_BOUND, MIDDAY_CHOP, VOLATILITY_COMPRESSION, NEWS_DISTORTED (too fast and erratic), NO_TRADE

Features:

Opening range size vs 5-day average (normalized)
Breakout bar's volume vs prior 5-bar average
Breakout bar close position (high-close = strong)
Distance from OR high/low at breakout
Gap direction alignment with breakout direction
Relative ATR at open
Type: Primarily rule-based with a confirmation gate. The rule defines the breakout; a simple logistic regression or rule-based confidence filter gates whether the breakout has quality.

Entry logic: Price breaks above OR high AND closes above it on a 5-minute bar, volume is above 1.5× prior 5-bar average, gap direction aligns (or no gap). Wait for first close above/below, then enter on next bar open.

Invalidation: Price immediately reverses back inside OR within 2 bars — abort. Volume absent on break.

Stop placement: At OR midpoint. Maximum stop = 1 ATR.

Target: 2× risk. This is a momentum entry with expected follow-through. Do not use partial exits — this model either works or doesn't.

Aggressiveness: Moderate-high. Build second. Dangerous in news-distorted opens.

SPEC-3: Failed Breakout Reversal Model
Purpose: Fade false breakouts — trade the trap. When price breaks a key level, runs stops, then reverses sharply back inside structure.

Allowed regimes: BREAKOUT_FAILURE, RANGE_BOUND (with key level violations), MEAN_REVERSION Forbidden regimes: TREND_UP, TREND_DOWN, BREAKOUT_CONTINUATION, OPENING_DRIVE, VOLATILITY_EXPANSION, NEWS_DISTORTED, NO_TRADE

Features:

Distance of initial breakout thrust (ATR)
Time duration of breakout before reversal
Volume spike at breakout vs subsequent bar volume
Distance from VWAP at reversal point
Speed of retracement vs speed of breakout
Wick-to-body ratio of breakout bar
Type: Rule-based with strict criteria. Do not use ML here initially — the setup is too specific and data will be sparse.

Entry logic: Price breaks prior high/low by 0.3–0.8 ATR, then closes back inside structure on same or next bar. Enter in reversal direction when price re-enters structure. Stop above the false breakout extreme.

Invalidation: Breakout exceeds 1.5 ATR from level (genuine break, not a trap). Volume remains elevated on reversal bar (indicates continuation, not exhaustion).

Stop placement: Just beyond the failed breakout extreme. Typically tight: 0.4–0.7 ATR.

Target: 1.5× risk. These moves resolve quickly or not at all.

Aggressiveness: Moderate. Build third. Most dangerous if misclassified — a real trend breakout looks identical to a false one for 1–2 bars.

SPEC-4: VWAP Mean Reversion Model
Purpose: Trade price returning to VWAP from extended deviation during range or chop days.

Allowed regimes: RANGE_BOUND, MEAN_REVERSION, MIDDAY_CHOP, VOLATILITY_COMPRESSION Forbidden regimes: TREND_UP, TREND_DOWN, BREAKOUT_CONTINUATION, OPENING_DRIVE, VOLATILITY_EXPANSION, NEWS_DISTORTED, NO_TRADE

Features:

Distance from VWAP in ATR units (z-score of deviation)
VWAP slope (should be flat or mildly sloped)
Number of VWAP crosses in prior 30 bars
Relative volume (should be low — high volume in range day signals something brewing)
Time of day (midday works best, avoid using in final 30 minutes)
ATR vs 5-day ATR ratio (should be below 1.0)
Type: Rule-based. VWAP mean reversion is a well-understood mechanical setup. Entry when |deviation| > 1.5 ATR, direction back toward VWAP.

Entry logic: Price deviates more than 1.5 ATR from VWAP, shows a reversal bar (wick in deviation direction, close toward VWAP), VWAP slope is near-flat, enter in direction of VWAP.

Invalidation: Volume spikes (regime is changing). Price reaches 2× ATR deviation without reversal. Regime flips to trend.

Stop placement: At 2× ATR from VWAP (or recent swing). Target: VWAP itself.

Aggressiveness: Conservative. Build fourth. Strong prop-firm profile: small, tight, high frequency, modest gains.

SPEC-5: Stand-Down / No-Trade Mode
Purpose: Explicitly do nothing. This is a first-class strategy module, not an absence of strategy.

Activated when: NEWS_DISTORTED, NO_TRADE, RISK_OFF, any regime with confidence < 0.35, post–stop-out within the first 60 minutes, after 3 consecutive losses, in final 15 minutes of session.

This module returns no signals. Its presence in the system forces the policy selector to have an explicit "do nothing" state rather than defaulting to the least-bad active specialist.

Build Order and Danger Assessment
Model	Build Order	Danger Level	Prop-Firm Fit
SPEC-1: Trend Pullback	First	Low	High
SPEC-2: ORB Confirmation	Second	Medium	Medium
SPEC-4: VWAP Mean Reversion	Third	Low	High
SPEC-3: Failed Breakout Reversal	Fourth	High	Medium
SPEC-5: Stand-Down	First (always)	None	Critical
Postpone for later:

Liquidity sweep model (requires DOM data, complex)
Momentum continuation (too similar to SPEC-1, adds correlation risk)
Any overnight / multi-session models (not day trading)
5. POLICY SELECTOR DESIGN
Role and Responsibilities
The policy selector is the arbiter. It takes regime probabilities, specialist model outputs, event flags, time-of-day state, and risk engine state, and produces exactly one of these outputs per bar:

ACTIVE(specialist_id, direction, size_multiplier): One specific specialist is permitted, with specified direction and sizing
REDUCED(specialist_id, direction, size_multiplier < 1.0): Same, but sized down
STANDDOWN: No trading permitted this bar
DEFENSIVE: Only micro contracts, only SPEC-4 (VWAP reversion) or no trade
Decision Logic
Step 1: Check hard blocks first

If risk engine returns SHUTDOWN → output STANDDOWN, stop
If event flag IMMINENT (< 30 min to major event) → STANDDOWN
If time is in final 15 minutes → STANDDOWN
If time is in first 15 minutes → STANDDOWN (pre-open noise)
Step 2: Evaluate regime confidence

If classifier_confidence < 0.35 → STANDDOWN
If classifier_confidence < 0.50 → DEFENSIVE mode (only SPEC-4 or none)
Step 3: Map primary regime to allowed specialists

Primary Regime	Allowed Specialists	Blocked Specialists
TREND_UP	SPEC-1 (long only), SPEC-2 (long only)	SPEC-3, SPEC-4
TREND_DOWN	SPEC-1 (short only), SPEC-2 (short only)	SPEC-3, SPEC-4
RANGE_BOUND	SPEC-4	SPEC-1, SPEC-2
BREAKOUT_CONTINUATION	SPEC-1, SPEC-2	SPEC-4
BREAKOUT_FAILURE	SPEC-3	SPEC-1, SPEC-2, SPEC-4
MEAN_REVERSION	SPEC-3, SPEC-4	SPEC-1, SPEC-2
VOLATILITY_EXPANSION	SPEC-5 (standdown)	ALL
VOLATILITY_COMPRESSION	SPEC-4 (reduced)	SPEC-1, SPEC-2, SPEC-3
OPENING_DRIVE	SPEC-2 only	SPEC-3, SPEC-4
MIDDAY_CHOP	SPEC-4 or none	SPEC-1, SPEC-2, SPEC-3
NEWS_DISTORTED	SPEC-5 (standdown)	ALL
RISK_OFF	SPEC-5 (standdown)	ALL
NO_TRADE	SPEC-5 (standdown)	ALL
Step 4: Resolve specialist signal quality

Each active specialist outputs a setup quality score (0–1)
If no specialist produces quality score > 0.55 → STANDDOWN
If highest quality score is 0.55–0.70 → allow with size multiplier 0.5
If highest quality score is 0.70–0.85 → allow with size multiplier 1.0
If highest quality score > 0.85 → allow with size multiplier 1.0 (never exceed 1.0 based on confidence alone)
Step 5: Conflict resolution

If two specialists both qualify → take the one with higher quality score only
Never allow two specialists simultaneously open
Never blend signals
Step 6: Anti-thrashing filter

Track last 3 policy outputs
If policy has changed 3+ times in 6 bars → force STANDDOWN for 3 bars
This prevents policy oscillation caused by regime probability instability
Step 7: Repeated failure penalty

If the active specialist has had 2 losses in the current session under the current regime → reduce size multiplier by 0.5
If 3 losses in current session under current regime → force SPEC-5 (standdown) for this regime for remainder of session

6. HARD RISK ENGINE DESIGN
Architecture Principle
The risk engine is a stateful session guardian. It maintains its own state independently of all models. No model output, no confidence score, no regime classification can override a hard limit. The risk engine speaks last and can always say no.

Account-Specific Parameters
Sub-$2,000 Personal Account
Parameter	Value	Rationale
Max risk per trade	$40 (2% of $2,000)	One bad trade cannot materially harm account
Max daily loss	$80 (4% of $2,000)	Protects from cascade failure
Max trades per day	4	Prevents overtrading under losses
Max consecutive losses	3	Pattern of bad trading → pause
Max open risk at once	$40 (one position only)	No pyramiding
Defensive mode trigger	-$50 daily loss	Reduces to half-size for remainder
Shutdown trigger	-$80 daily loss	End session
Gain protection trigger	+$60 daily gain	Risk-off: reduce to minimum viable size
Prop Firm Account (~$2,000 Drawdown Limit)
Parameter	Value	Rationale
Max risk per trade	$50 (2.5% of $2,000 limit)	Firm's daily loss limit is real
Max daily loss	$100 (5% of $2,000 limit)	Far inside firm's limit
Max trades per day	5	Slightly more room
Max consecutive losses	3	Same
Max open risk at once	$50	Same — one position
Defensive mode trigger	-$60 daily loss	Earlier than personal account
Shutdown trigger	-$100 daily loss	Hard stop
Gain protection trigger	+$80 daily gain	Protect gains aggressively
Weekly loss limit	-$200	Prevent runaway weekly drawdown
Risk States
NORMAL        → All specialists active per policy, full size allowed
REDUCED       → Policy can activate specialists but size multiplier capped at 0.5
DEFENSIVE     → Only SPEC-4 or standdown; micros only; size multiplier = 0.25
PAUSED        → No new trades for N bars (timed pause, not day-end)
SHUTDOWN      → No new trades for rest of session; exit existing positions
Risk Reduction Triggers
After drawdown:

At 50% of daily loss limit → REDUCED mode
At 75% of daily loss limit → DEFENSIVE mode
At 100% of daily loss limit → SHUTDOWN
After consecutive losses:

2 consecutive losses → size multiplier × 0.75
3 consecutive losses → PAUSED (30 minutes), then REDUCED for rest of session
If pause completes and regime has changed favorably → resume REDUCED
If pause completes and regime unchanged → SHUTDOWN
After volatility events:

If ATR exceeds 2× 5-day average ATR → force REDUCED or DEFENSIVE
If any single bar's range exceeds 3× typical bar range → PAUSED (10 minutes)
Immediately after major event release → PAUSED (15 minutes minimum)
Gain protection:

Once daily gain exceeds the gain protection trigger: new risk limit = half the daily gain at that moment
Example: Gained $80, protection triggers. New stop-loss from this point = $40. If account gives back $40 from peak, SHUTDOWN.
Instrument-specific blocks:

CL/MCL: Only permitted in NORMAL or REDUCED state. Blocked entirely in DEFENSIVE.
GC/MGC: Permitted in all states (lower volatility profile relative to crude)
NQ/MNQ: Permitted in REDUCED; blocked in DEFENSIVE on high-volatility days
ES/MES: Permitted in all risk states
Preventing Model Confidence Override
This is enforced structurally, not by instruction. The risk engine receives only:

Current daily P&L
Current position
Current consecutive loss count
Current time
Current ATR ratio
Current event flag
It does NOT receive model confidence scores. It cannot be influenced by them. It operates purely on observable, auditable facts.

Preventing Revenge Trading
The system cannot feel revenge, but it can exhibit systematic re-entry after losses in the same direction. Prevention:

After 3 consecutive losses → PAUSED (time-locked, not overridable)
After PAUSED ends → max trades for session is reduced by 2
After SHUTDOWN → the session is over. No re-entry, no exceptions.
Detecting Model Underperformance
Track rolling performance metrics in Python:

Last 10 trades: win rate and average P&L
Current week: win rate and average P&L
Expected win rate from backtesting by regime
If current 10-trade win rate < 0.35 AND average P&L is negative → force DEFENSIVE for rest of session.
If current week win rate < 0.30 → flag for human review; force REDUCED for following session.

System Behavior
Before major events (Tier 1 and 2):

30+ minutes before: SPEC-2 and SPEC-1 are suppressed (trend-following into news = stop-run risk)
SPEC-4 with extremely tight stops may still operate if in a clear range
Position size caps at 50% of normal
All open positions must be flat by 15 minutes before Tier 1 events
Immediately after major events (0–20 minutes):

Forced STANDDOWN regardless of regime or specialist quality
Do not try to trade the initial spike — slippage during the first post-event minutes is brutal
Risk engine sets post-event flag that persists for 20 minutes minimum
Rest of day after abnormal event-driven movement:

If the post-event move is > 2× expected range for that event → flag NEWS_DISTORTED regime
NEWS_DISTORTED forces SPEC-5 (standdown) for a minimum of 60 minutes
After 60 minutes, allow REDUCED risk re-engagement only if regime classifier shows stabilizing signal (VWAP convergence, declining ATR slope)

Load gap context at session open
Tag as "gap day" for full session

Use the Economic Calendar from Investing.com or ForexFactory scraped daily (legal and free). Parse into the JSON format above. Update the local file before each session. That requires ML pipelines (NLP) that are far beyond what this system needs at this stage.

9. MODELING APPROACH BY LAYER
A. Regime Classifier: LightGBM Multiclass Classifier
Recommendation: LightGBM with probability output

LightGBM's advantages for this task:

Trains on 6–12 months of 5-minute bar data in seconds
Handles 15–25 features without normalization
Produces calibrated predict_proba output across all 13 classes
Built-in regularization protects against overfit
Retraining is fast enough to do daily
Why not HMM: HMMs assume Markov structure (current state depends only on prior state) and Gaussian emissions. Real market regimes have long-memory properties and non-Gaussian distributions. HMMs are elegant but empirically underperform gradient boosting on this task.

Why not neural nets: Sample efficiency is far worse; interpretability suffers; retraining is slower; no clear benefit over LightGBM on tabular features.

Label construction: Assign regime labels in hindsight to each 5-minute bar based on end-of-session analysis. Label a bar as "TREND_UP" if the session from that bar forward showed continuous higher highs/lows and the session closed significantly above open. Labels require judgment — some manual review of training data is essential.

B. Specialist Models: Rule-Based with Optional LightGBM Quality Scorer
Recommendation: Rule-based setups gated by LightGBM binary classifier (quality score)

Do not attempt to build full ML entry models initially. Instead:

Define the setup rules clearly (see Section 4)
When a rule-based setup is detected, compute a feature vector for that specific setup instance
Run the feature vector through a small LightGBM binary classifier trained to predict "did this specific setup produce a profitable outcome within the next N bars"
Output is a quality score (0–1)
Policy selector uses the quality score to gate execution
This hybrid is more trainable than pure ML (you need the rules to reduce the universe of potential setups) and more adaptive than pure rules (the quality scorer adjusts to current feature distributions).

Training signal: Label each rule-based setup instance as 1 (profitable) or 0 (not profitable) based on forward outcome within 6 bars or 1× ATR excursion.

C. Policy Selector: Deterministic Rule Engine
Recommendation: Hard rule-based state machine

Do not use ML for the policy selector. The policy selector needs to be:

Auditable (you must be able to explain every decision)
Stable (no model drift)
Predictable (no unexpected outputs)
Fast (no inference latency)
A rule-based state machine with the decision table from Section 5 is the correct tool. The ML is inside the specialist quality scorers and regime classifier — the policy selector is the controlled bridge between them.

If you later want to optimize policy parameters (quality thresholds, size multipliers), use a structured parameter search over backtested outcomes — not online learning.

D. Risk Engine: Hardcoded Arithmetic Rules
Recommendation: Pure deterministic code with no ML component whatsoever

The risk engine must be:

Deterministic
Stateful
Impossible to override by model output
Auditable and testable with unit tests
Use a Python class with explicit state transitions and hard arithmetic limits. Test it exhaustively. The risk engine is the most critical component in the entire system — it is not a machine learning problem.

Communication Specification
Transport: TCP socket, loopback (127.0.0.1:5678), one persistent connection per session

Protocol: Length-prefixed JSON. Each message is prefixed with a 4-byte little-endian integer indicating the length of the following JSON payload. This eliminates the fragmentation problem of raw TCP.

Flow:

NT8 sends request (length prefix + JSON)
Python reads, processes, responds (length prefix + JSON)
Repeat every bar close
Timeout handling:

NT8 waits maximum 2 seconds for response
If timeout: NT8 enters safe mode (no new entries, manage existing stops only)
Python side: if no request received for 10+ minutes during RTH → log warning
Failure handling:

If Python server crashes: NT8 detects disconnected socket, retries connection every 5 seconds, enters safe mode
If NT8 crashes: Python server keeps listening, cleans session state on reconnect handshake
If NT8 reconnects mid-session: Python server receives reconnect handshake with session_id + current session P&L to sync state
Latency requirement: Target < 50ms per request/response cycle on loopback. LightGBM inference on 25 features is sub-millisecond. Total processing including all layers should be < 10ms. Socket round-trip on loopback is < 1ms. Margin is very comfortable.

Reconnect Handshake Schema
NT8 sends on connection:

json
{
  "message_type": "CONNECT_HANDSHAKE",
  "session_id": "20260404_MNQ",
  "session_date": "2026-04-04",
  "instrument": "MNQ",
  "session_pnl": -32.50,
  "open_position_size": 0,
  "consecutive_losses": 2
}
Python responds:

json
{
  "message_type": "CONNECT_ACK",
  "session_id": "20260404_MNQ",
  "risk_state": "REDUCED",
  "reconnected": true,
  "prior_session_trades": 2
}

12. TRAINING / VALIDATION / RETRAINING PIPELINE
Raw Data Requirements
5-minute OHLCV bars: minimum 18 months of clean data per instrument
1-minute OHLCV bars: for feature context
Daily OHLCV bars: for overnight context features
Economic calendar: event dates, times, tiers (free from Investing.com or FRED)
Session open/close timestamps (RTH and ETH)
Data source: NT8 historical export, NinjaTrader Data, or Databento / Norgate (best quality)
Quality requirements:

Zero forward-leaking data
Adjust for daylight saving time session boundaries
Remove clearly erroneous bars (spike filters)
Handle holidays and early closes explicitly
Labeling Design
Regime labels: Applied in hindsight at end of session. A human or rule-based labeler reviews each session's bar-by-bar price action and assigns regime context per 5-minute bar. This takes 1–2 hours per month of data. It is worth doing carefully — garbage labels produce garbage models.

Specialist quality labels: Applied per individual setup instance. For each bar where a rule-based setup fires, label the outcome within the forward window (6 bars or 1× ATR, whichever comes first). Label 1 if the trade reached 1× ATR profit without hitting stop. Label 0 otherwise.

Train/Validation/Test Design
Use a strict walk-forward design — no shuffling, no cross-validation across time.

Training data:    months 1–12 (in-sample)
Validation data:  months 13–15 (offline tuning)
Test data:        months 16–18 (untouched until final evaluation)
Production:       rolling window, retrain monthly
Walk-forward retraining:

Retrain monthly on the most recent 12 months of data
Validate on the most recent 2 months (out-of-sample)
Deploy if out-of-sample performance meets minimum criteria
Log all retraining events with performance metrics
When to Retrain
Retrain monthly on a fixed schedule for the regime classifier. Retrain specialist quality scorers when their rolling 30-trade performance drops below baseline. Retrain when feature drift is detected: If the distribution of any top-5 most important feature shifts by more than 1.5 standard deviations from training distribution, flag for review.

When NOT to Retrain
Do not retrain after 1–2 bad weeks (could be regime anomaly, not model decay)
Do not retrain based on a single event day (news distortion)
Do not retrain to "fix" the model's performance on the most recent 10 days — this is overfitting
Do not retrain without reviewing the new training data for quality issues
Regime Drift Monitoring
Track the distribution of classified regimes over rolling 20-session windows. If the frequency of any regime shifts materially (> 2 standard deviations from 6-month historical frequency), flag for review. This can signal:

Real regime shift in the market → possibly appropriate to retrain
Classifier drift → possibly a data quality issue
Session anomaly → wait for more data
Model Decay Detection
Track the following metrics per 10-trade rolling window:

Specialist quality score vs actual outcome correlation
Regime classifier accuracy on labeled test periods
Daily P&L per regime
If quality score correlation drops below 0.15 (essentially random), the quality scorer is no longer useful. Force rules-only mode and schedule immediate retraining.

15. FAILURE MODES AND KILL SWITCHES
1. Regime Classifier Overfitting
Why it happens: Labels are assigned in hindsight; features leak information about the future; training data is too short. Danger level: HIGH — wrong regime → wrong specialist → systematic losses. Detection: Out-of-sample regime accuracy degrades; specialist models lose money in regimes they should win in. Kill switch: If specialist win rate under a specific regime drops below 35% over 20 trades, suppress that specialist for that regime permanently until retrained.

2. Unstable Regime Flipping
Why it happens: Classifier uncertainty near regime boundaries causes rapid oscillation. Danger level: MEDIUM — causes policy thrashing, multiple position changes, overtrading costs. Detection: Anti-thrashing filter fires more than 3 times per session. Kill switch: Force minimum 3-bar regime persistence. If flipping persists → raise confidence threshold to 0.65 for that session.

3. Specialist Model Operates in Wrong Regime
Why it happens: Policy selector logic error, regime classification error, edge case not covered in decision table. Danger level: HIGH — model designed for one condition deployed in opposite condition. Detection: Log every trade with the regime at entry time. If SPEC-1 fires during RANGE_BOUND → immediate audit. Kill switch: Policy selector must perform explicit regime-specialist compatibility check before every approval. Incompatible combination → hard reject regardless of quality score.

4. Policy Thrashing
Why it happens: Low regime confidence → rapid oscillation between specialists or between active and standdown. Danger level: MEDIUM — operational chaos, position whipsawing. Detection: More than 3 policy state changes in 6 bars. Kill switch: Anti-thrashing filter forces STANDDOWN for 3 bars when detected.

5. Event-Day Misclassification
Why it happens: Pre-event calm → classifier sees low volatility → classifies as RANGE or COMPRESSION → specialists try to enter right before a CPI print. Danger level: VERY HIGH — can produce catastrophic slippage. Detection: Event calendar must override classifier output. If event_imminent = true → STANDDOWN regardless of regime. Kill switch: Event calendar check runs FIRST in the pipeline, before regime classifier output reaches policy selector. The calendar is a hard override, not a soft suggestion.

6. NT8/Python Communication Failure
Why it happens: Python crashes, network hiccup, garbage JSON, timeout. Danger level: MEDIUM — system cannot trade intelligently without Python. Detection: NT8 timeout on socket receive. Kill switch: NT8 enters SAFE MODE: no new entries, actively manage existing position stops, alert human. NT8 should NEVER trade without Python confirmation — the risk of random entries without intelligent oversight is worse than missing trades.

7. Slippage Spikes
Why it happens: Trading during news, low liquidity, bid-ask spread expansion. Danger level: MEDIUM-HIGH — small accounts can lose an entire trade's value to slippage. Detection: Track fill price vs order price. If slippage > 0.5 ATR on 3+ trades → flag. Kill switch: If average slippage exceeds 30% of expected stop distance → force session review. Block certain regimes (VOLATILITY_EXPANSION, NEWS_DISTORTED) where slippage risk is highest.

8. Data Feed Problems
Why it happens: NT8 data feed disconnects, stale bars, data gaps. Danger level: HIGH — stale features → wrong signals. Detection: Feature validation layer checks for stale timestamps, impossible values (zero ATR, zero volume), and bar gaps. Kill switch: If feature vector fails validation → Python returns BLOCK. Never trade on potentially stale data.

9. Inference Lag
Why it happens: Python server overloaded, model becomes too large, network congestion. Danger level: LOW-MEDIUM — delays entry, may miss bar close execution window. Detection: Log latency for every request. Alert if > 100ms. Kill switch: Timeout = 2 seconds. If exceeded → BLOCK. The trade is not worth the uncertainty.

10. Correlated Losses Across Instruments
Why it happens: Trading NQ and ES simultaneously in same direction during a correlated sell-off; both models lose simultaneously. Danger level: HIGH — doubles drawdown velocity. Kill switch (preventive): Only trade one instrument at a time in V1 and V2. If multi-instrument trading is added in V3, impose a correlated-loss detection rule: if both instruments are in losing positions simultaneously → close the worse performer immediately.

11. False Confidence During Unusual Regimes
Why it happens: Model trained on normal market conditions sees an unusual regime (RISK_OFF, flash crash, circuit breaker) and confidently misclassifies it. Danger level: HIGH — confident wrong classification is worse than low-confidence classification. Detection: Monitor for ATR ratio > 2.5 (extreme volatility outlier). This almost always means unusual conditions. Kill switch: If ATR ratio > 2.0 → force DEFENSIVE. If ATR ratio > 3.0 → force SHUTDOWN regardless of classifier output.

# Python Intelligence Server — Per-Request Processing

def handle_request(request: dict) -> dict:
    
    # Step 0: Validate incoming features
    features = feature_store.validate_and_preprocess(request)
    if features is None:
        return build_block_response("INVALID_FEATURES")
    
    # Step 1: Get event context (FIRST — hard override)
    event_ctx = event_calendar.get_event_flags(request["timestamp"])
    if event_ctx["event_imminent"]:
        return build_block_response("EVENT_BLACKOUT")
    
    # Step 2: Check session phase (hard override)
    session_phase = get_session_phase(request["timestamp"])
    if session_phase in ["PRE_OPEN", "FIRST_15", "LAST_15", "AFTER_CLOSE"]:
        return build_block_response("SESSION_PHASE_BLOCK")
    
    # Step 3: Run regime classifier
    regime_output = regime_classifier.classify(features)
    
    # Step 4: Run allowed specialists
    specialist_outputs = {}
    for specialist_id in get_allowed_specialists(regime_output):
        specialist = get_specialist(specialist_id)
        specialist_outputs[specialist_id] = specialist.evaluate(
            features, regime_output
        )
    
    # Step 5: Policy selector
    policy_output = policy_selector.select_policy(
        regime_output=regime_output,
        specialist_outputs=specialist_outputs,
        event_ctx=event_ctx,
        session_phase=session_phase,
        risk_state=risk_engine.get_current_state()
    )
    
    # Step 6: Risk engine gate (LAST — cannot be overridden)
    risk_output = risk_engine.check_permission(
        policy_output=policy_output,
        instrument=request["instrument"]
    )
    
    # Step 7: Build response
    response = build_response(policy_output, risk_output, regime_output)
    
    # Step 8: Log everything
    logger.log_decision(request, response, regime_output, 
                        specialist_outputs, policy_output, risk_output)
    
    return response

class RiskEngine:
    def __init__(self, account_config):
        self.account_type = account_config["type"]  # "personal" or "prop"
        self.max_risk_per_trade = account_config["max_risk_per_trade"]
        self.daily_loss_limit = account_config["daily_loss_limit"]
        self.max_trades_per_day = account_config["max_trades_per_day"]
        self.max_consecutive_losses = account_config["max_consecutive_losses"]
        
        # Session state — reset daily
        self.session_pnl = 0.0
        self.session_high_water = 0.0
        self.session_trades = 0
        self.consecutive_losses = 0
        self.risk_state = "NORMAL"
        self.paused_until = None
        self.shutdown = False
        self.gain_protection_active = False
        self.gain_protection_floor = None

    def session_reset(self):
        """Call at session open"""
        self.session_pnl = 0.0
        self.session_high_water = 0.0
        self.session_trades = 0
        self.consecutive_losses = 0
        self.risk_state = "NORMAL"
        self.paused_until = None
        self.shutdown = False
        self.gain_protection_active = False
        self.gain_protection_floor = None

    def update_pnl(self, trade_pnl: float):
        """Call after every closed trade"""
        self.session_pnl += trade_pnl
        self.session_trades += 1
        
        if trade_pnl < 0:
            self.consecutive_losses += 1
        else:
            self.consecutive_losses = 0
            
        if self.session_pnl > self.session_high_water:
            self.session_high_water = self.session_pnl
        
        # Update gain protection
        gain_trigger = account_config["gain_protection_trigger"]
        if (not self.gain_protection_active and 
                self.session_pnl >= gain_trigger):
            self.gain_protection_active = True
            self.gain_protection_floor = self.session_pnl / 2
        
        self._evaluate_state()

    def _evaluate_state(self):
        """Update risk state based on current session metrics"""
        
        # Check shutdown conditions first
        if self.session_pnl <= -self.daily_loss_limit:
            self.shutdown = True
            self.risk_state = "SHUTDOWN"
            return
            
        if (self.gain_protection_active and 
                self.session_pnl <= self.gain_protection_floor):
            self.shutdown = True
            self.risk_state = "SHUTDOWN"
            return
        
        # Check pause condition
        if self.consecutive_losses >= self.max_consecutive_losses:
            self.paused_until = now() + timedelta(minutes=30)
            self.risk_state = "PAUSED"
            return
        
        # Check defensive threshold
        if self.session_pnl <= -(self.daily_loss_limit * 0.75):
            self.risk_state = "DEFENSIVE"
            return
        
        # Check reduced threshold
        if self.session_pnl <= -(self.daily_loss_limit * 0.50):
            self.risk_state = "REDUCED"
            return
        
        # If paused period expired, transition to REDUCED
        if (self.risk_state == "PAUSED" and 
                self.paused_until and now() > self.paused_until):
            self.risk_state = "REDUCED"
            self.paused_until = None
            return
        
        # Default to NORMAL if none of above triggered
        if self.risk_state not in ["PAUSED", "REDUCED", "DEFENSIVE"]:
            self.risk_state = "NORMAL"

    def check_permission(self, policy_output, instrument, atr_ratio) -> dict:
        
        self._evaluate_state()
        
        # Hard ATR ratio checks
        if atr_ratio > 3.0:
            return {"trade_permitted": False, "reason": "ATR_EXTREME_VOLATILITY",
                    "risk_state": "SHUTDOWN"}
        
        if atr_ratio > 2.0 and self.risk_state == "NORMAL":
            self.risk_state = "DEFENSIVE"
        
        # Hard checks
        if self.shutdown:
            return {"trade_permitted": False, "reason": "SESSION_SHUTDOWN",
                    "risk_state": self.risk_state}
        
        if self.risk_state == "PAUSED":
            return {"trade_permitted": False, "reason": "SYSTEM_PAUSED",
                    "risk_state": self.risk_state}
        
        if self.session_trades >= self.max_trades_per_day:
            return {"trade_permitted": False, "reason": "MAX_TRADES_REACHED",
                    "risk_state": self.risk_state}
        
        if policy_output["no_trade_flag"]:
            return {"trade_permitted": False, "reason": "POLICY_STANDDOWN",
                    "risk_state": self.risk_state}
        
        # Compute size multiplier
        base_multiplier = policy_output.get("size_multiplier", 1.0)
        risk_multiplier = {
            "NORMAL": 1.0,
            "REDUCED": 0.5,
            "DEFENSIVE": 0.25,
        }.get(self.risk_state, 0.0)
        
        final_multiplier = base_multiplier * risk_multiplier
        
        return {
            "trade_permitted": True,
            "risk_state": self.risk_state,
            "size_multiplier": final_multiplier,
            "max_risk_this_trade": self.max_risk_per_trade * risk_multiplier,
            "daily_pnl": self.session_pnl,
            "consecutive_losses": self.consecutive_losses
        }

public class PythonBridgeClient
{
    private TcpClient _client;
    private NetworkStream _stream;
    private readonly string _host = "127.0.0.1";
    private readonly int _port = 5678;
    private readonly int _timeoutMs = 2000;
    private bool _safeMode = false;
    
    public bool IsConnected => _client?.Connected ?? false;
    
    public void Connect()
    {
        try {
            _client = new TcpClient();
            _client.ReceiveTimeout = _timeoutMs;
            _client.SendTimeout = _timeoutMs;
            _client.Connect(_host, _port);
            _stream = _client.GetStream();
            _safeMode = false;
            SendHandshake();
        }
        catch (Exception ex) {
            EnterSafeMode($"Connection failed: {ex.Message}");
        }
    }
    
    public DecisionResponse RequestDecision(FeatureVector features)
    {
        if (!IsConnected) {
            TryReconnect();
            if (!IsConnected) return SafeModeResponse();
        }
        
        try {
            // Build request JSON
            var request = BuildRequestJson(features);
            var requestBytes = Encoding.UTF8.GetBytes(request);
            
            // Send length prefix (4 bytes, little-endian) + JSON
            var lengthPrefix = BitConverter.GetBytes(requestBytes.Length);
            _stream.Write(lengthPrefix, 0, 4);
            _stream.Write(requestBytes, 0, requestBytes.Length);
            
            // Read response with timeout
            var lengthBuf = new byte[4];
            ReadExact(_stream, lengthBuf, 0, 4);
            int responseLength = BitConverter.ToInt32(lengthBuf, 0);
            
            var responseBuf = new byte[responseLength];
            ReadExact(_stream, responseBuf, 0, responseLength);
            
            var responseJson = Encoding.UTF8.GetString(responseBuf);
            _safeMode = false;
            return ParseResponse(responseJson);
        }
        catch (Exception ex) {
            EnterSafeMode($"Communication error: {ex.Message}");
            return SafeModeResponse();
        }
    }
    
    private void EnterSafeMode(string reason)
    {
        _safeMode = true;
        LogMessage($"SAFE_MODE_ACTIVATED: {reason}");
        // NT8 strategy will see SafeModeResponse and halt new entries
    }
    
    private DecisionResponse SafeModeResponse()
    {
        return new DecisionResponse {
            Permission = "BLOCK",
            Reason = "PYTHON_DISCONNECTED_SAFE_MODE"
        };
    }
    
    private void TryReconnect()
    {
        // Attempt reconnect once; strategy handles retry timing
        try { Connect(); } catch { }
    }
}

18. PRACTICALITY FILTER (Self-Challenge)
Here is an honest evaluation of this design's weakest points:

Too Complex (Simplify or Defer)
13 regime classes is too many to label reliably. Start with 6: TREND_UP, TREND_DOWN, RANGE_BOUND, BREAKOUT_CONTINUATION, BREAKOUT_FAILURE, and NO_TRADE. Collapse the others into these categories for V1. Add finer granularity later when you have labeled data quality to support it.

Quality scorers for specialist models are premature in V1. Rule-based setups with fixed quality thresholds are sufficient to start. The LightGBM quality scorer requires sufficient labeled setup instances — you may not have enough from the first few months. Replace quality scorer with a simpler scoring rubric (integer score based on how many conditions are met) for V1.

Per-specialist rolling win rate tracking in the policy selector adds statefulness that is easy to get wrong. Defer the "suppress specialist after 3 losses in current regime" logic until V2. In V1, just use the simpler "pause after 3 consecutive losses" logic in the risk engine.

The anti-thrashing filter is important but easy to implement incorrectly. In V1, replace it with a simpler rule: minimum 3 bars between policy changes.

Most Likely to Overfit
Regime classifier labels are subjective and hindsight-biased. The labeling process is the highest-risk part of the whole system. If you label a day as "TREND_UP" because it closed 50 points above open, but the first 60 minutes were chop, your classifier will learn to activate trend-following strategies during opening chop. Review every label manually for the first 60 days of training data.

The specialist quality scorer will overfit if the training set is small. LightGBM needs 500+ positive and 500+ negative examples per specialist to produce reliable probability estimates. If you have fewer, use logistic regression with L2 regularization instead — it generalizes better on small samples.

Fragile in Live Trading
TCP socket communication is more fragile than file-based communication for non-critical latency cases. File-based polling is slower (1–5 second decision cycle) but is nearly indestructible. If the 5-minute decision cadence is acceptable for your strategy (it usually is for bar-based day trading), consider starting with file-based communication in V1 and switching to sockets in V2. This removes an entire failure mode from the first live deployment.

Event calendar must be updated daily before the session. This is a manual operational step. If you forget to update it and a CPI print lands without the flag set, the system will trade through it. Automate the calendar update or add a redundant manual check.

NinjaTrader 8 can behave unexpectedly during connection errors, data feed issues, and order routing problems. The local emergency stop in NT8 (section 11) is essential. Do not rely solely on Python to limit risk during platform instability.

Unrealistic for NT8
Full VWAP computation in NinjaTrader 8 is straightforward but requires session-anchored VWAP, not bar-anchored. Make sure the FeatureCalculator indicator uses session-anchored VWAP explicitly.

Real-time volume delta (buy-side vs sell-side) requires Level 2 data and a CME subscription. Remove this from the feature list for any trader without that subscription. All features listed in the Regime Classifier section above are achievable with standard NT8 data.

// How NT8 requests permission before placing every trade

private bool RequestTradePermission(string direction, out DecisionResponse response)
{
    response = null;
    
    // Step 1: Build current feature vector
    FeatureVector features = BuildFeatureVector();
    
    // Step 2: Validate feature vector locally (sanity check)
    if (!ValidateFeatures(features)) {
        Log("Feature validation failed — blocking trade", LogLevel.Warning);
        return false;
    }
    
    // Step 3: Send request to Python
    DecisionResponse pythonResponse = pythonBridge.RequestDecision(features);
    
    // Step 4: Null/error check
    if (pythonResponse == null) {
        Log("Null response from Python — blocking trade", LogLevel.Warning);
        return false;
    }
    
    // Step 5: Check permission field
    if (pythonResponse.Permission != "EXECUTE" && 
        pythonResponse.Permission != "EXECUTE_REDUCED") {
        Log($"Python blocked trade: {string.Join(", ", pythonResponse.RejectionReasons)}",
            LogLevel.Information);
        return false;
    }
    
    // Step 6: Direction alignment check
    if (pythonResponse.Direction != direction) {
        Log($"Direction mismatch: requested {direction}, " +
            $"Python approved {pythonResponse.Direction} — blocking", 
            LogLevel.Warning);
        return false;
    }
    
    // Step 7: Local hard limit check (belt-and-suspenders)
    double stopDistanceDollars = pythonResponse.StopDistancePoints 
                                 * TickSize * BigPointValue 
                                 / TickSize;
    if (stopDistanceDollars > LocalMaxRiskPerTrade) {
        Log($"Stop distance ${stopDistanceDollars:F2} exceeds local limit " +
            $"${LocalMaxRiskPerTrade:F2} — blocking", LogLevel.Warning);
        return false;
    }
    
    // Step 8: Already in position?
    if (Position.MarketPosition != MarketPosition.Flat) {
        Log("Already in position — ignoring new entry signal", LogLevel.Information);
        return false;
    }
    
    // Step 9: All checks passed
    response = pythonResponse;
    Log($"Trade APPROVED: {direction} via {pythonResponse.ActiveSpecialist}, " +
        $"size×{pythonResponse.FinalSizeMultiplier:F2}", LogLevel.Information);
    return true;
}
